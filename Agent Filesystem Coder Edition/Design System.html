the situation room,
codified.

A drop-in design system for the AFS console & marketing surfaces. Tokens first, components second, principles always. Pair tokens.css with the styled-components theme adapter (§12) and your existing routes will inherit the situation-room aesthetic without rewriting markup.

[ 01 // PRINCIPLES ]
design tenets — the rules of engagement
06 ITEMS

These principles arbitrate every decision below. When a pattern conflicts with a principle, the principle wins.

01.
Operators, not tourists.

The audience is engineers running production systems. Density > whitespace. Information > decoration. Every pixel earns its keep.

02.
Hard edges. Real shadows.

Square corners by default. Rounding (max 4px) reserved for form controls and pills. No drop-shadows for depth — use rule lines and surface tone.

03.
Mono is the display face.

Headlines and chrome are JetBrains Mono. Sans is reserved for body prose. The terminal is the brand — let it speak.

04.
One signal color.

Lime (--afs-accent) is the only accent. State colors (warn/err/info) are functional, not decorative. Resist the gradient.

05.
Live, not loud.

Motion = blink, pulse, ticker. All slow, all purposeful. The room is operational, not flashy. Respect prefers-reduced-motion.

06.
Everything is observable.

Surfaces should reveal state — checkpoints, ops/sec, agent counts. Static UI is suspect. If it's running, show it running.

[ 06 // MOTION ]
durations, easings, signals — slow & purposeful
5 DURATIONS · 4 EASINGS

Motion is signal, not flair. Three keyframes — afs-blink, afs-pulse, afs-spark — handle ~95% of feedback. UI transitions sit at --afs-dur-fast (120ms) on --afs-ease. Always honors prefers-reduced-motion.

@keyframes afs-blink
1400ms · steps(2,end)
@keyframes afs-pulse
2000ms · ease-out
@keyframes afs-spark
1600ms · ease-in-out
/* durations */
--afs-dur-instant: 80ms;     /* state flips */
--afs-dur-fast:    120ms;    /* default ui */
--afs-dur-base:    180ms;    /* sidebar collapse */
--afs-dur-slow:    280ms;    /* modal in/out */
--afs-dur-tick:    1400ms;   /* led blink cadence */

/* easings */
--afs-ease:     cubic-bezier(.2,.8,.2,1);   /* default */
--afs-ease-in:  cubic-bezier(.4,0,1,1);
--afs-ease-out: cubic-bezier(0,0,.2,1);
--afs-ease-step: steps(2,end);             /* terminal-cursor blink */

[ 10 // VOICE & COPY ]
how AFS talks
3 RULES
01.
Lowercase chrome.

Nav, buttons, captions, tickers — all lowercase. Sentence case for prose. Caps-lock reserved for classification stamps and section eyebrows.

02.
Operator vocabulary.

"Workspace", "fork", "checkpoint", "agent", "incident", "ops/sec" — terms an SRE recognizes. Avoid marketing words: "seamless", "powerful", "leverage".

03.
Telegram cadence.

Short. Specific. Numeric. "1.4ms p50 read latency." not "blazing fast performance." Headlines can be poetic; chrome cannot.

04.
Wink, don't shout.

"a filesystem for ~~humans~~ agents" works. "Revolutionize your AI stack!" does not. The brand has a sense of humor — dry, military, internal.

[ 11 // DO / DON'T ]
guardrails
SHIPPING CHECKS
// DO
  • Use the lime accent once per primary action per surface.
  • Default to square corners. Round controls only (max 4px).
  • Pair every state pill with an LED dot when it's "live".
  • Show numbers — ops/sec, p50, count — not vague verbs.
  • Anchor sections with [ NN // SECTION_NAME ] eyebrows.
  • Keep dividers thin (1px, --afs-line) — never strong fills.
// DON'T
  • Don't introduce a second accent. Lime is the only signal.
  • Don't gradient backgrounds. Use surface tones (bg/bg-1/bg-2).
  • Don't drop-shadow cards. Use rule borders for separation.
  • Don't animate decoratively. Motion = signal (blink/pulse).
  • Don't use sans for chrome. Mono is the brand voice.
  • Don't put more than one ticker on a page — it's a pulse, not noise.

[ 12 // ENGINEERING HANDOFF ]
applying this to agent-filesystem
3 STEPS

The repo uses styled-components with a theme prop. Add tokens.css as a global stylesheet, then expose the same values on the styled-components theme object so existing styled components inherit them via ${({theme})=>…} without rewrites.

// STEP 1 · DROP TOKENS.CSS INTO ui/src

// ui/src/main.tsx
import "./tokens.css";        // FIRST — before component CSS
import "./index.css";

// STEP 2 · ADAPTER FOR styled-components

// ui/src/foundation/theme/afs-theme.ts
export const afsTheme = {
  semantic: {
    color: {
      bg:        "var(--afs-bg)",
      panel:     "var(--afs-bg-1)",
      ink:       "var(--afs-ink)",
      accent:    "var(--afs-accent)",
      text: { neutral400: "var(--afs-ink-dim)" },
      // …
    },
  },
  font: { mono: "var(--afs-font-mono)", sans: "var(--afs-font-sans)" },
  space: (n: number) => `var(--afs-sp-${n})`,
  radius: { none: "var(--afs-r-0)", sm: "var(--afs-r-1)", md: "var(--afs-r-2)" },
};

// STEP 3 · WRAP THE TREE

// ui/src/main.tsx
import { ThemeProvider } from "styled-components";
import { afsTheme } from "./foundation/theme/afs-theme";

<ThemeProvider theme={afsTheme}>
  <RouterProvider router={router} />
</ThemeProvider>

After this, the existing sidebar.styles.ts, app-bar.styles.ts, and styled rules referencing theme.semantic.color.text.neutral400 will resolve to the AFS tokens automatically. The sidebar already references --afs-muted, --afs-line, --afs-accent, --afs-panel directly — those map 1:1 to tokens.css; just rename the variables (or add aliases) and the cutover is clean.
